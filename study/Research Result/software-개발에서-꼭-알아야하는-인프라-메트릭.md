# 핵심 요약
- 시스템 메트릭은 인프라 수준과 애플리케이션 수준으로 나뉘며, 서비스의 건강 상태를 객관적으로 판단하는 기준이 된다.
- 인프라 메트릭의 핵심 4요소는 CPU 사용량, 메모리 사용량, 네트워크 대역폭, 디스크 사용량 및 I/O이다.
- 각 메트릭은 하드웨어(트랜지스터, 커패시터, NIC 등)의 물리적 동작 원리와 밀접하게 연결되어 있다.
- 메트릭의 '정상' 범위는 절대적이지 않으며, 운영 경험을 통해 팀 내에서 합의하고 관리해 나가야 한다.

# 상세 내용

### 1. CPU Usage (CPU 사용률)
- **원리**: CPU 내 수십억 개의 트랜지스터가 1초 동안 얼마나 활발하게 0과 1을 스위칭(연산)했는지를 나타내는 백분율이다.
- **영향**: 사용률이 100%에 도달하면 정해진 시간 내 처리 가능한 연산량을 초과하게 된다. 이로 인해 요청이 대기열에 쌓이고 결국 타임아웃(Timeout)이 발생한다.

### 2. Memory Usage (메모리 사용률)
- **원리**: 데이터를 전기적으로 저장하는 커패시터(Capacitor)에 얼마나 많은 데이터가 적재되어 있는지를 나타낸다.
- **OOM(Out Of Memory)**: 물리적 메모리와 스왑 공간이 모두 소진되면, 커널은 시스템 전체 마비를 막기 위해 'OOM 킬러'를 가동하여 특정 프로세스를 강제 종료한다.
- **성능 저하**: 메모리가 부족하여 OS가 디스크를 메모리처럼 사용하는 스왑(Swap)을 시작하면 성능이 수천 배 느려질 수 있다.

### 3. Network Bandwidth (네트워크 대역폭)
- **원리**: NIC(네트워크 인터페이스 카드)가 1초 동안 송수신할 수 있는 비트(0과 1)의 양이다.
- **영향**: 대역폭이 한계에 도달하면 패킷 손실이 발생하거나 지연 시간(Latency)이 급증한다. CPU나 메모리가 충분하더라도 네트워크 대역폭 부족으로 병목 현상이 생길 수 있다.

### 4. Disk Usage & I/O (디스크 사용량 및 입출력)
- **Disk Usage**: 저장 장치에 데이터가 저장된 물리적 공간을 의미하며, 100% 도달 시 쓰기 작업이 실패하여 애플리케이션 오류가 발생한다.
- **Disk I/O**: 디스크와 메모리 간의 데이터 전송 활동이다. 읽기/쓰기 빈도가 너무 높으면 응답 시간이 느려지고 시스템 부하가 증가한다.

# 인사이트 / 할 일
- **인사이트**: 인프라 메트릭을 단순히 숫자로만 보지 말고, 하드웨어 자원의 한계가 소프트웨어의 타임아웃이나 크래시로 이어지는 인과관계를 이해하는 것이 중요하다.
- **인사이트**: 커널의 OOM 킬러나 세그먼테이션 폴트는 시스템을 보호하기 위한 최후의 방어 기제임을 인지해야 한다.
- **할 일**: 현재 운영 중인 서비스의 '정상 상태' 메트릭 범위를 정의하고, 임계치 설정 시 하드웨어 사양과 비즈니스 요구사항을 대조해 볼 것.
- **할 일**: 로그 분석 시 OOM 발생 여부를 체크하고, 주기적인 디스크 I/O 모니터링을 통해 데이터베이스나 로그 저장소의 병목을 사전에 파악할 것.
