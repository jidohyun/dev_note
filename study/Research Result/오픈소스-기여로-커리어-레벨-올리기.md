# 핵심 요약
- 오픈소스 기여는 단순한 취미가 아니라 글로벌 수준의 코드 품질을 경험하고 실력을 증명하는 강력한 커리어 브랜딩 수단이다.
- 기여 전략은 업무 효율을 높이는 '업무 관련 기술'과 지속 가능성을 확보하는 '좋아하는 기술' 두 가지 방향으로 접근할 수 있다.
- 메인테이너의 사고 비용을 줄여주는 구조화된 PR(Pull Request) 작성이 빠른 머지(Merge)와 협업 신뢰도의 핵심이다.
- 머지 이후에는 사내외 공유와 이력서 업데이트를 통해 기여 경험을 실질적인 커리어 자산으로 전환해야 한다.

# 상세 내용

1. 오픈소스 기여의 가치
- 글로벌 수준의 코드 리뷰: 전 세계 숙련된 개발자들로부터 깊이 있는 피드백을 받으며 설계 능력을 향상할 수 있다.
- 전문가 브랜딩: GitHub 기여 내역은 이력서보다 강력한 신뢰를 주며, 채용 과정에서 코딩 테스트 면제나 기술 자문 등 다양한 기회로 이어진다.

2. 시작을 위한 두 가지 전략
- 전략 A (업무 관련 기술): 현재 회사에서 사용하는 기술에 기여하는 방식. 학습 내용이 실무에 즉시 반영되어 ROI(투자 대비 효율)가 가장 높고 진입 장벽이 낮다.
- 전략 B (좋아하는 기술): 개인적으로 애정을 가진 라이브러리에 기여하는 방식. 내재적 동기가 강해 꾸준히 활동할 수 있으며, 사용자 관점에서 가치 있는 개선안을 제안하기 유리하다.

3. 기여의 단계별 프로세스
- 이슈 발견: 버그, 문서(README 오타 등), UX/DX 개선점 등 사소한 불편함에서 시작한다.
- 이슈 등록 vs 바로 PR: 아키텍처 수정 등 논의가 필요한 대규모 작업은 이슈를 먼저 등록하고, 오타 수정이나 명백한 버그는 바로 PR을 보낸다.
- 협업 규칙: Fork -> Clone -> Branch -> Edit -> PR 순서를 따르며, 작업 전 댓글로 의사를 밝혀 작업 중복을 방지한다.

4. 빠르게 머지되는 PR 작성법 (사고 비용 최소화)
- BACKGROUND: 문제의 맥락과 변경이 필요한 이유 설명.
- WHAT CHANGED: 수정 사항 요약.
- WHY: 특정 접근 방식을 선택한 엔지니어링적 판단 근거.
- ALTERNATIVES: 고려했던 다른 대안과 채택하지 않은 이유.
- IMPACT: 하위 호환성 유지 여부 등 변경이 미치는 영향.
- TESTING: 검증 방법 및 결과 제시.

5. 기여를 커리어 자산화하는 방법
- 사내 공유: "기능 구현자"를 넘어 라이브러리 내부 구조를 이해하는 "기술 전문가"로 포지셔닝한다.
- 대외 공유: 문제 해결 과정을 서사적으로 풀어 블로그나 SNS에 기록하여 해당 도메인의 전문성을 노출한다.
- 이력서 업데이트: "어떤 문제를 해결하여 어떤 임팩트를 주었는지" 정량적이고 구체적인 성과로 기술한다.

# 인사이트 / 할 일
- 오픈소스 기여는 완벽한 준비가 되었을 때 하는 것이 아니라, 업무 중 발견한 작은 불편함을 해결하는 것부터 시작된다는 점을 명심하자.
- 현재 프로젝트에서 사용 중인 핵심 라이브러리의 GitHub 저장소를 방문하여 'Good First Issue' 라벨이 붙은 이슈가 있는지 확인해보기.
- 향후 PR을 보낼 때 본문에 포함된 6단계 템플릿(Background, What, Why, Alternatives, Impact, Testing)을 적용하여 메인테이너와의 소통 효율 높이기.
- 문서 오타 수정이나 번역 개선처럼 진입 장벽이 낮은 것부터 실천하여 '기여자(Contributor)'로서의 경험 쌓기.
