# 핵심 요약
- Next.js 13부터 도입된 `error.js`는 파일 시스템 기반으로 페이지 및 전역 에러를 체계적으로 관리할 수 있게 함.
- 전역 에러를 처리하는 `global-error`와 특정 경로의 에러를 처리하는 `error`로 구분하여 유연한 대응이 가능함.
- React의 `ErrorBoundary`와 Next.js의 `error.js`를 조합하여 컴포넌트 단위부터 페이지 단위까지 계층적인 에러 핸들링 시스템을 구축할 수 있음.
- 에러 발생 시 `reset()` 함수를 제공하여 페이지 전체 새로고침 없이 특정 영역의 상태만 초기화하고 재시도할 수 있는 사용자 경험(UX)을 제공함.

# 상세 내용
- **Next.js error.js의 개념 및 종류**
  - `global-error`: 애플리케이션 전체에서 발생하는 치명적인 에러를 처리하며, 루트 레이아웃까지 감싸는 최상위 경계임.
  - `error`: 특정 세그먼트(경로)와 그 하위 페이지에서 발생하는 에러를 처리함.

- **동작 방식**
  - Next.js는 서버 시작 시 각 경계의 `error.tsx`를 찾아 `AppRouter` 내부의 `ErrorBoundary`로 구성함.
  - 에러가 발생하면 해당 에러는 가장 가까운 상위 `ErrorBoundary`로 전파되어 처리됨.
  - 클라이언트 컴포넌트로 작성되어야 하며, `error` 객체와 `reset` 함수를 props로 받음.

- **React ErrorBoundary와의 차이점**
  - **Next.js error.js**: 파일 시스템 기반으로 페이지 단위의 에러를 관리함. 특정 경로 전체의 UI를 대체할 때 유리함.
  - **React ErrorBoundary**: 코드 내에서 특정 컴포넌트를 직접 감싸서 사용함. 위젯이나 버튼 등 아주 작은 단위의 UI 예외 처리에 효과적임.

- **실무적인 에러 핸들링 전략**
  - **통합 접근**: 페이지 수준의 큰 에러는 `error.js`로 관리하고, 페이지 내부의 독립적인 컴포넌트(예: 사이드바, 추천 목록 등)는 React `ErrorBoundary`로 감싸서 에러가 페이지 전체로 퍼지는 것을 방지함.
  - **사용자 경험 최적화**: 에러 발생 시 무조건적인 에러 페이지 노출보다는, 문제가 발생한 부분만 대체 UI를 보여주고 나머지 기능은 정상 작동하게 유지하는 것이 중요함.

# 인사이트 / 할 일
- **에러 범위의 최소화**: 에러가 발생했을 때 서비스 전체가 중단되는 것이 아니라, 문제가 생긴 컴포넌트만 격리하여 사용자 이탈을 막는 설계가 필요함.
- **재시도 로직 구현**: `error.js`에서 제공하는 `reset()` 함수를 활용해 사용자가 스스로 에러 상황을 복구할 수 있는 '다시 시도' 버튼을 반드시 배치해야 함.
- **액션**: 현재 진행 중인 프로젝트에서 모든 에러를 공통 에러 페이지로 보내고 있지는 않은지 점검하고, 독립적인 UI 요소들에 `ErrorBoundary`를 적용해보기.
