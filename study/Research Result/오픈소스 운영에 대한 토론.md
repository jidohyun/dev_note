## 배경

창희님이 민희 님과 오픈소스 운영에 대한 토론을 진행했고 이때 남긴 노트를 LLM을 이용해서 요약했습니다.

> 오픈소스 운영의 핵심은  
> “코드를 잘 만드는 것”이 아니라  
> “개념을 이해시키고, 쓰게 만들고, 다시 돌아오게 하는 것”이다.

## 노트

### 1. 오픈소스를 하는 이유와 원동력

- 오픈소스는 쓰는 사람이 늘어날수록 기여자도 늘어나는 구조
    - 특히 사용자가 곧 프로그래머인 도메인에서는 더 그렇다.
- 개인적 동기만으로는 한계가 있고, “많이 쓰이게 만드는 것” 자체가 가장 강력한 성장 엔진
- hackers.pub, fedify 등은 단순 구현체가 아니라 개념(ActivityPub, Fed, CRDT)을 알리기 위한 수단

### 2. 기여자는 어떻게 생기는가

초기 현실

- 대부분의 오픈소스는 초기엔 1인 개발
- 컨트리뷰터가 생기기까지는 시간 + 사용자 수 누적이 필요

사람들이 기여하지 않는 이유

- 개념을 모름 (CRDT, ActivityPub 자체를 모른다)
- 개념의 매력을 모름
- 도구(Yorkie/Fedify)를 보면 무엇을 할 수 있는지 감이 안 옴

→ 그래서 “코드”보다 먼저 해야 할 일은 이해와 설득

### 3. 오픈소스 마케팅 = 교육 + 반복 노출

핵심 전략

- 글을 많이 쓴다
- “이걸 쓰면 바닥부터 구현 안 해도 된다”
- “이 문제를 이렇게 단순하게 해결할 수 있다”
- CRDT / Fedify 같은 생소한 개념을 계속 풀어서 설명
- Yorkie/Fedify를 개념의 구체적인 구현 예시로 제시
    - [https://fedify.dev/tutorial/microblog](https://fedify.dev/tutorial/microblog)

홍보 채널

- Hacker News, Reddit, Lobsters 적극 활용
- 릴리즈마다 반드시 공유
- 처음엔 지인 업보트 요청도 효과 큼 (0 vs 1은 큰 차이)

### 4. 릴리즈 노트 & 체인지로그 운영 철학

- changelog만 쓰지 말 것
- 릴리즈 노트에는 반드시:
- 이번 릴리즈의 하이라이트
- [https://github.com/fedify-dev/fedify/discussions/462](https://github.com/fedify-dev/fedify/discussions/462)
- [https://github.com/fedify-dev/fedify/blob/main/CHANGES.md#version-190](https://github.com/fedify-dev/fedify/blob/main/CHANGES.md#version-190)

왜 이 기능이 필요한지 (이슈 → 해결)

구조:

- Overview: 왜 중요한가
- Release Note: 무엇이 바뀌었나
- Changelog: 상세 변경
- 기여자 이름 언급 → 동기 강화
- “계속 릴리즈가 올라온다” = 살아있는 프로젝트라는 신호
- 대부분 AI로 작성 (LLM 친화적)

### 5. 문서가 곧 커뮤니티의 핵심

Fedify 사례에서 중요한 포인트

1. Full-featured 튜토리얼

- 보름 가까이 풀타임으로 튜토리얼 제작
- 튜토리얼의 각 단계 = 실제 커밋
- 거의 복붙만으로 완성 가능
- 스크린샷, 코드, 디버깅 방법까지 포함

👉 결과: “문서를 따라가다 보면 개념이 자연스럽게 이해됨”

2. 유저 테스트 기반 문서

- 지인에게 직접 튜토리얼을 시켜봄
- 막히는 지점 관찰 → 문서 개선
- 홈페이지도 “만든 사람 시점”이 아니라
- “처음 쓰는 사용자 시점”으로 재검토

3. 문서 철학

- internal 문서보다 컨셉 문서
- 테스트, 디버깅 문서까지 제공
- 프론트엔드 + DB까지 예제 제공
- TS / JSX 문법 설명도 아낌없이 포함
- verbose해 보여도 디버깅 시 큰 가치
- 참고한 문서 스타일: SQLAlchemy 공식 문서
- 문서 정적 사이트 생성기
    - [https://github.com/vuejs/vitepress](https://github.com/vuejs/vitepress)
    - [https://github.com/withastro/starlight](https://github.com/withastro/starlight)
    - [https://github.com/web-infra-dev/rspress](https://github.com/web-infra-dev/rspress)

### 6. LLM 시대를 고려한 문서 전략

문서는 LLM 친화적이어야 함

- 좋은 사례:
    - bun.sh의 Copy as Markdown
    - llms.txt 제공 (bun, fedify)
- context7 같은 곳에 문서 인덱싱
- [agents.md](http://agents.md/) 표준화 → 잘 써두면 좋음
- 문서와 코드가 같은 레포에 있는 구조 선호
- 기여자에게도 docs 기여를 적극 권장
